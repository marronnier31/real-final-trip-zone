서비스 개요

TripZone

고객이 실제로 예약하고 운영자가 바로 관리하는 여행 숙박 플랫폼

TripZone는 숙소 탐색, 예약, 결제, 리뷰, 쿠폰, 운영 관리까지 한 흐름으로 이어지는 3-role 서비스입니다.

User Seller Admin Booking Coupon Dashboard
3 핵심 역할

고객, 판매자, 관리자 흐름을 한 서비스 안에서 구현

1 통합 여정

탐색부터 결제, 운영 반영까지 하나의 데모 라인

Live 관리형 프로모션

이벤트와 쿠폰이 관리자 등록 후 즉시 사용자 화면 반영

Verified 실행 검증

예약 생성, 결제 표시, 이벤트/쿠폰 노출 흐름까지 확인

Connected Flow

핵심은 각 화면이 아니라 실제 서비스 흐름이 끝까지 이어진다는 점입니다.

E2E

탐색 → 예약 → 결제 → 마이페이지 → 운영 반영

01
고객 시점 실사용 흐름을 기준으로 화면을 설명
02
운영 시점 판매자와 관리자가 같은 데이터를 운영
03
검증 시점 직접 생성한 예약과 프로모션 결과를 근거로 제시

Why It Matters

숙소 플랫폼은 기능이 아니라 연결이 끊기면 바로 신뢰를 잃습니다.

1
예약 실패 날짜, 객실, 결제, 요청사항 중 하나라도 어긋나면 고객 이탈
2
혜택 불일치 관리자가 등록한 쿠폰이 사용자 화면에 안 보이면 운영 신뢰도 하락
3
운영 단절 판매자 예약관리, 관리자 이벤트관리, 고객 마이페이지가 서로 맞지 않으면 서비스 완성도가 무너짐

Signal

Customer

예약은 됐는데 결제 정보가 안 보임

상세 화면에서 결제 수단이 누락되면 구매 신뢰가 즉시 깨집니다.

Promotion

관리자 쿠폰이 사용자 목록에서 식별 안 됨

등록은 되어도 노출 방식이 다르면 운영 성과를 설명할 수 없습니다.

Admin

등록 실패 안내가 틀린 위치에 표시

운영자가 원인을 못 보면 실제 서비스 운영도 막힙니다.

Key Insight

핵심은 화면 수보다 동일한 데이터를 끝까지 이어주는 연결 품질이었습니다.

고객 화면이 좋아 보여도 운영 데이터와 연결되지 않으면 서비스 신뢰가 약합니다.
Half
운영 화면이 많아도 사용자 경험으로 이어지지 않으면 서비스성이 약합니다.
Half
같은 데이터가 고객 · 판매자 · 관리자 화면에서 일관되게 보일 때 구현 완성도와 서비스 완성도가 같이 올라갑니다.
Full
핵심은 “기능이 있다”가 아니라 “등록한 데이터가 실제 사용자 여정에 반영된다” 입니다.

Snapshot

Booking

요청사항 저장, 결제 수단 표시, 중복 날짜 실패 사유 노출

Coupon

관리자 등록값이 사용자 `/events`에 바로 노출

Admin

등록 실패 사유를 패널 안에 표시해 운영성 강화

Seller

예약관리 패널에서 요청메시지까지 확인 가능

Structure

TripZone는 3-role이 같은 운영 라인을 공유하는 구조로 설계했습니다.

User

탐색 · 예약 · 결제

숙소 탐색 후 예약, 결제, 쿠폰 적용, 마이페이지 확인까지 연결

Seller

숙소 · 예약 운영

판매자 대시보드에서 예약 상태, 요청사항, 판매 지표 확인

Admin

이벤트 · 쿠폰 · 관리

관리자가 등록한 프로모션이 사용자 화면과 DB에 일관되게 반영

Structure

Booking

예약 흐름은 “실패 원인 표시”와 “성공 후 결제 반영”까지 닫았습니다.

A
실패 원인 노출 겹치는 날짜에서는 서버 에러를 가리지 않고 실제 메시지를 사용자에게 보여줍니다.
B
결제 수단 매핑 정렬 프론트 결제 옵션 값을 백엔드 enum과 맞춰 카드/간편결제/가상계좌 흐름을 정렬했습니다.
C
마이페이지 상세 정합성 예약번호와 결제 데이터 연결 기준을 맞춰 결제 수단과 날짜를 상세 화면에 노출합니다.

Validation

실검증 결과

예약 생성 성공 후 `/my/bookings/272` 이동, 결제 정보와 요청사항이 상세 화면에 반영됨을 확인했습니다.

POST /api/booking 200 POST /api/payments 200 신용/체크카드 표시

Promotion

이벤트와 쿠폰은 관리자 입력값이 실제 고객 화면에 보이도록 정리했습니다.

Admin

등록

이벤트와 쿠폰을 운영 기간, 상태, 할인 값 기준으로 관리

Frontend

노출

`/events`에서 관리자 쿠폰명과 할인 정보가 함께 식별되도록 변경

Validation

반영

등록 직후 사용자 목록에 실데이터 기준으로 즉시 표시되는지 검증

운영자 입장에서는 “등록 가능”이 아니라 “등록 후 사용자 화면에서 바로 보인다”가 핵심입니다.

Validation

실검증 결과

`QA 퍼센트 쿠폰`, `QA 쿠폰 저장 재검증`, `QA 이벤트 20260407`가 `/events`에 반영된 상태를 확인했습니다.

POST /api/coupon 200 GET /api/coupon/list 200 coupon name visible

Operations

운영 화면은 단순 CRUD가 아니라 문제를 바로 판단할 수 있게 다듬었습니다.

1
쿠폰 등록 폼 단순화 정률 기준으로 정리하고 날짜 입력 구조를 더 명확하게 재구성했습니다.
2
실패 문구 위치 수정 헤더 뒤로 숨던 실패 메시지를 우측 등록 패널 안으로 이동했습니다.
3
날짜 역전 사전 차단 종료 일시가 시작보다 빠를 때 서버 실패처럼 보이지 않도록 프론트에서 먼저 막습니다.

Quality

운영 완성도 포인트

관리자는 등록이 안 될 때 이유를 즉시 보고, 정상 등록 시 바로 목록과 사용자 화면까지 확인할 수 있습니다.

패널 내 오류 표시 날짜 역전 검증 등록 성공 토스트

Evidence

추정이 아니라 실제 검증 로그를 근거로 정리했습니다.

Booking

예약 생성 성공

`POST /api/booking => 200`, `POST /api/payments => 200`, 예약 상세 진입까지 확인

Promotion

이벤트/쿠폰 즉시 노출

관리자 등록 후 `/events` 목록에서 실데이터 카드 확인

Database

user/admin 컬럼 저장 확인

`EVENTS.ADMIN_USER_NO`, `COUPONS.ISSUED_BY_USER_NO` 실제 저장 확인

Build

프론트 빌드 검증

최종 수정 후 빌드 통과, chunk warning만 존재

Result

결과적으로 TripZone는 “보여주는 서비스”가 아니라 “이어지는 서비스”가 되었습니다.

01 고객 관점에서 예약, 결제, 마이페이지 흐름을 실제로 데모할 수 있습니다.
02 운영 관점에서 관리자 이벤트/쿠폰 등록이 사용자 화면과 DB까지 연결됩니다.
03 기능 개수보다 데이터 정합성과 검증 증거를 분명하게 설명할 수 있습니다.
04 고객 경험과 구현 완성도를 동시에 보여줄 수 있습니다.

Summary

TripZone의 핵심 메시지는 간단합니다.

고객이 예약한 정보가 마이페이지에 남고,
관리자가 등록한 쿠폰이 사용자 화면에 보이며,
판매자가 실제 예약 요청을 운영할 수 있어야 합니다.
